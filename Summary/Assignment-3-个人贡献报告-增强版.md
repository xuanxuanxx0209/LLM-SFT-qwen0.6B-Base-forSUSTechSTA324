### 3.3 第二阶段:训练策略探索与多轮消融

**工作内容**

为继续提升模型表现,围绕"数据组成 × 训练超参"这两条主线,进行了系统性的消融实验:

| 探索方向 | 具体操作 | 结论 |
|---|---|---|
| **加入通用领域数据** | 混入 `general_data_200.jsonl`(共 5 个领域:编程 / 生产力 / 生活 / 社会 / 文本)或 100 条子集,得到 `final_en_sft_scoring_aligned_plus_general100.jsonl` 等数据集 | loss 收敛正常,但验证集分数无明显提升;通用数据稀释了数学样本占比 |
| **加入 `\boxed{none}` 拒答样本** | 使用 `total.jsonl`(2243 条,含 706 条 `\boxed{none}`),后随机降采样得 `total_filtered.jsonl`(1645 条,保留 100 条 none + 8 条空回答,seed=42) | 模型对"无解题"判别略有响应,但整体准确率反而下降 |
| **加入批判性思考样本** | `sftdata_plus_general100_plus_critical_thinking100.jsonl`,bs=16(4×4),epoch=3 | 同上,无稳定增益 |
| **学习率消融** | `5e-6`、`2e-6`、`1e-6` 三档对比 | `5e-6` 收敛最快但易过拟合;`1e-6` 收敛不充分;`2e-6` 介于两者之间 |
| **优化器系数** | `adam_betas=(0.9, 0.95)` vs `(0.95, 0.999)` | 影响有限 |
| **梯度累积** | `ga=2` vs `ga=4` vs `ga=8` | 等效 batch 16–32,差异不显著 |
| **正则项** | grad clip + L2 + warmup ratio 调整 | 对最终评测无突破性影响 |

**关键结论**:在仅依赖单一全量 SFT 的范式下,该数据规模上模型性能已逼近收益上限,需要切换到**更精细的训练策略**(分阶段训练 + 局部精修)才能继续提升。这一阶段虽然没有产出新的最佳模型,但**为后续 LoRA 思路的提出与团队策略转型提供了关键的负面证据**。

### 3.4 第三阶段:基于 V3 的 LoRA 弱项强化

在缪梓航同学提出"高比例基础题 + 低比例进阶题"的训练数据组织思路并训练出 `qwen3-0.6B-mathsft-V3`(验证集整体准确率 44.30%)之后,本人承接了在该基模上进一步提升的工作。

#### 3.4.1 V3 模型分题型能力分析

**工作内容**

- 使用 `run_eval_with_dump.py`(基于 vLLM,bf16,`max_model_len=4096`,温度 0,1000 题约 6–8 分钟跑完),对 V3 进行批量推理,输出逐题结果文件 `eval_dumps/V3_per_item.jsonl`;
- 按验证集自带的 `perturbation_type` 字段进行分桶统计,识别出 V3 的弱项题型。


**关键发现**:V3 在 **critical thinking** 题型上完全失败(`0/130`,占整体 13 个百分点)。这类题是把 GSM8K 题目中的关键数字删掉变成"无解题",期望答案为 `None`。V3 因为没见过这种"应当拒答"的样本,会强行编出一个数。

#### 3.4.2 LoRA 训练数据集构建(`build_lora_dataset.py`)

**工作内容**

- 数据来源:数据组清洗的 `/home/ubuntu/jgy-dataset/7460_cleaned.jsonl`(7456 条干净池);
- 编写 `build_lora_dataset.py`,采用 **"少量 CT 合成题 + 大量 replay 基础题"** 的混合配方:
  - 从 shuffle 后的池中前 500 条作为 critical thinking 合成题候选,用于教模型识别"题目缺失关键信息时应输出 `None`";
  - 从 `pool[500:]` 起依次取 replay 样本(基础题),作为锚点防止灾难性遗忘;

- 两轮迭代:
  - `lora_weakness_train.jsonl`:**150 CT + 400 replay = 550 条**(CT 占比 27%,后被验证为 CT 比例过高)
  - `lora_weakness_train_v2.jsonl`:**100 CT + 700 replay = 800 条**(CT 占比 12.5%),为后续主力训练数据集

#### 3.4.3 LoRA 微调:6 个版本迭代,v5 胜出



**版本演进逻辑**:
- **v1 → v2**:v1 学习率 `1e-4` 偏大、CT 占比 27% 过高,导致灾难性遗忘(整体反而下降 3.6%);v2 把学习率降到 `5e-5`,replay 比例提升至 87.5%,首次实现稳定正向收益。
- **v2 → v3 → v4**:在 `rank=8` 框架内尝试 epoch 与 CT 数的微调,均在 +0.2 ~ +0.6 区间波动,提示在该 rank 下已接近瓶颈。
- **v4 → v5**:**降低 rank 到 4**(alpha 同比减半到 8),令 LoRA 容量进一步缩小,起到更强的正则化作用,意外地实现了**全题型同时上升** —— 在 CT、digit expansion、numerical substitution 上分别取得 **+11.54%、+5.08%、+4.09%** 的题型级提升,整体准确率达到 **46.80%**,成为小组最佳。
- **v5 → v6**:验证学习率假设(`2.5e-5`,v5 的一半),收敛偏慢且综合表现回落到 43.90%,**反向确认 v5 的超参组合为局部最优**。

## 4. 个人反思与总结

### 4.1 主要贡献回顾

回顾整个期中项目,本人在微调环节的核心贡献可以概括为四点:

1. **建立了从基模到第一个超 baseline 模型(j1 / model-j)的端到端训练流水线**,验证了项目技术路线的可行性,使小组在评测平台上取得首次有效提交;
2. **第二阶段的瓶颈期通过大量负面实验排除若干无效方向**,为团队转向"分阶段训练 + LoRA 精修"的策略提供了实证基础;
3. **基于 V3 基模独立完成了从能力分析、数据筛选、LoRA 训练到合并评测的完整闭环**,产出小组当前综合表现最好的模型 `V3-lora-merged-v5`,**整体准确率 46.80%,相对 V3 baseline 绝对提升 +2.50%**;

### 5.2 遇到的主要问题与解决方法

- **问题 1:第二阶段反复尝试无明显增益,一度陷入瓶颈。**
  - 解决:从单纯调超参转向重新审视数据组成与训练范式,通过逐题 dump 分析 V3 的弱项题型,发现 `critical thinking` 题型 0/130 全错的关键短板,引入 LoRA 这种"局部精修"手段,跳出全量 SFT 的瓶颈。
- **问题 2:LoRA v1 直接灾难性遗忘 —— 整体准确率反而比 V3 baseline 下降 3.6%。**
  - 原因诊断:学习率 `1e-4` 过大,且 CT 题占比 27% 过高,导致模型把 V3 已学到的数学解题能力"擦掉"了。
  - 解决:把学习率降到 `5e-5`(降一个数量级),CT 占比降到 12.5%,replay 占比升至 87.5%,从 v2 起即恢复正向收益。
- **问题 3:LoRA 容易"按下葫芦浮起瓢" —— 弱项提升的同时强项回退。**
  - 解决:① 通过 `target_modules` 显式**排除 `lm_head` 与 `embed_tokens`**,降低对输出层先验的扰动;② 在 v5 中将 rank 进一步降到 4(alpha=8),用更小的 LoRA 容量起到正则化作用,使弱项提升的同时不破坏 V3 已有能力 —— 这是 v5 能在 CT、digit expansion、numerical substitution 等多个题型上**同时**取得正向提升的核心原因。

### 5.3 对团队协作的客观评价

本组的分工较为清晰:杨皓博同学负责数据清洗与质量把控,缪梓航同学负责数据清洗方案设计与执行，协助完成模型参数调优与训练工作，包括"高比例基础题"的全量 SFT 训练并产出 V3 基模,本人负责第一阶段全量 SFT、第二阶段策略探索与第三阶段 LoRA 微调。三者形成"**数据 → 基模 → 精修**"的清晰流水线,各环节产出可独立验证、互不重叠。

### 5.4 收获

本次项目让我对**小参数量模型在限定数据下的微调策略**有了更系统的认识:

- 全量 SFT 适合奠定整体能力基础,而 **LoRA 在已有较强基模时是更可控、更不易破坏既有能力的精修工具**;
- **数据的"针对性"远比数据的"数量"更重要** —— 模型的较大提升均源于较小数据集;
- **每一次评测必须是逐题可追溯的** —— 没有 `V3_per_item.jsonl` 这样的逐题 dump,我无法发现 CT 0/130 这个核心短板,后续的 LoRA 攻坚也就无从谈起;
- **更小的 LoRA rank 不一定更弱** —— 在已有较强基模时,过大的 rank 反而会破坏已有能力。v5(rank=4)优于 v2/v3/v4(rank=8)就是这个直觉的反例验证。
