## 2. 个人贡献总览

在本期中项目中,我承担的核心任务是**模型微调(Model Fine-tuning)**这一环节,具体包括:

1. **项目启动期**:在项目初期,深入研读老师发布的项目要求与评测规则,并向小组成员系统讲解任务目标、提交规范、评测平台使用方式与可行的技术路线,推动小组对任务边界达成统一理解。

3. **第一阶段 —— 全量 SFT 基线模型**:在拿到第一个高质量数据集 `jgy-dataset/final_en_sft_scoring_aligned.jsonl` 后,使用 OpenRLHF 框架对 `Qwen3-0.6B-Base` 进行**全参数 SFT 微调**,产出小组提交至评测平台的第一版相对 baseline 有提升的模型(评测平台版本号 **j1**,对应模型权重 `model-j.safetensors`)。
4. **第二阶段 —— 训练策略探索**:依据课堂上老师提到的训练改进思路,系统性尝试多种方案:加入通用领域数据(general data)、加入 boxed 答案为 `none` 的样本以增强模型对"无解/拒答"的判别能力,并对学习率、batch size、epoch、warmup 等超参数进行了大量消融实验。
5. **第三阶段 —— 基于队友新基模的 LoRA 微调**:在队友缪梓航同学提出"高比例基础题 + 低比例进阶题"的训练数据组织思路并训练出新一阶段的全量 SFT 模型 `qwen3-0.6B-mathsft-V3` 之后,我在该模型基础上承接了**LoRA 微调**的工作:
   - 对 V3 模型在验证集 1000 条上的逐题输出进行**分题型能力分析**;
   - 从队友清洗后的数据集中**针对性地选取 V3 模型表现较弱题型**的训练样本,构建专用 LoRA 训练集 `lora_weakness_train.jsonl` / `lora_weakness_train_v2.jsonl`;
   - 围绕 LoRA rank(r4 / r8 / r16)、学习率、epoch、目标模块等维度进行**5 个版本的迭代**;
   - 最终产出当前小组综合表现最好的模型 **`qwen3-0.6B-mathsft-V3-lora-merged-v5`**。
6. **评测与日志整理**:基于 vLLM 编写推理脚本 `run_eval.py` / `run_eval_lora_merged.py` / `run_eval_with_dump.py`,对每一版模型在验证集上的输出进行批量推理与逐题打分,生成分题型 dump 日志(`eval_dumps/V3_lora_merged_v*_per_item.jsonl`),为下一次迭代提供量化依据。

### 3.3 第二阶段:训练策略探索与多轮消融

**工作内容**

为继续提升模型表现,我系统地进行了以下尝试:

| 尝试方向 | 数据集 / 操作 | 结果 |
|---|---|---|
| 加入通用领域数据 | `final_en_sft_scoring_aligned_plus_general100.jsonl` 等(混入 100 条通用数据) | loss 收敛正常,但验证集分数无明显提升 |
| 加入 boxed=none 的样本 | 用于增强"无解/拒答"判别 | 部分题型表现略有变化,但整体未带来稳定增益 |
| 加入批判性思考样本 | `sftdata_plus_general100_plus_critical_thinking100.jsonl` | 同上 |
| 学习率消融 | `5e-6 → 2.5e-5` 等多组实验 | 学习率过大导致灾难性遗忘,过小则收敛不充分 |
| epoch / batch 调整 | 多组配置组合 | 无法突破第一阶段的瓶颈 |

**结论**:在仅依赖单一全量 SFT 的范式下,模型在该小数据规模上已逼近收益上限,需要切换到更精细的训练策略才能继续提升。这一阶段虽然没有产出新的最佳模型,但**为后续 LoRA 思路的提出提供了关键的负面证据与方向判断**。

#### 3.4.1 V3 模型的分题型能力分析

**工作内容**

#### 3.4.3 LoRA 微调:5 个版本的迭代

**工作内容**

- 训练脚本:`/home/ubuntu/run_lora_math_sft.py`(本人编写,基于 OpenRLHF + DeepSpeed,启用 LoRA 适配器)
- 合并脚本:`/home/ubuntu/merge_lora.py`(将 LoRA adapter 合并回基模,产出独立 safetensors)
- 关键 LoRA 配置(以最终选定版本 v5 为例):
  - `lora_rank=8`,`lora_alpha=16`,`lora_dropout=0.05`
  - `target_modules`: `q_proj, k_proj, v_proj, o_proj, gate_proj, up_proj, down_proj`(显式排除 `lm_head` 与 `embed_tokens`,避免破坏 V3 已学到的输出分布)
  - 学习率 `5e-5`,`max_epochs=2`,`max_len=4096`

**5 个版本的迭代逻辑**:

| 版本 | 主要变化 | 验证集表现 |
|---|---|---|
| v2 | 基线 LoRA(rank=8, lr=5e-5, epoch=2),首个稳定收敛版本 | 相对 V3 有提升 |
| v3 | 调整训练数据比例 / 训练步数 | 部分题型回退 |
| v4 | 进一步精炼弱项数据集 | 部分题型提升,部分回退 |
| **v5** | **数据 + 超参的最优组合(当前最佳)** | **综合表现最好,各题型均衡提升** |
| v6 | 学习率减半(2.5e-5)的消融实验 | 收敛偏慢,综合不如 v5,确认 v5 为最优点 |

最终选定 **`qwen3-0.6B-mathsft-V3-lora-merged-v5`** 作为小组提交的最终模型。

## 5. 个人反思与总结

### 5.1 主要贡献回顾

回顾整个期中项目,本人在微调环节的核心贡献可以概括为三点:

1. **建立了从基模到第一个超 baseline 模型(j1)的端到端训练流水线**,验证了项目技术路线的可行性;
2. **在第二阶段的瓶颈期,通过大量负面实验排除了若干无效方向**,为团队转向"分阶段训练 + LoRA 精修"的新策略提供了实证基础;
3. **基于 V3 基模独立完成了从能力分析、数据筛选、LoRA 训练到合并评测的完整闭环**,产出了小组目前综合表现最好的模型 `V3-lora-merged-v5`。

### 5.2 遇到的主要问题与解决方法

- **问题 1:第二阶段反复尝试无明显增益,一度陷入瓶颈。**
  - 解决:从单纯调超参转向重新审视数据组成与训练范式,通过逐题 dump 分析弱项,引入 LoRA 这种更"局部精修"的手段,跳出全量 SFT 的局限。
- **问题 2:LoRA 微调容易"按下葫芦浮起瓢" —— 弱项提升的同时强项回退。**
  - 解决:通过 `target_modules` 显式排除 `lm_head` 与 `embed_tokens`,降低对输出层先验的扰动;同时在训练数据中保留少量基础题作为锚点,使弱项提升的同时不破坏 V3 已有能力。这一思路在 v5 版本上得到验证。
- **问题 3:多版本实验如何高效对比。**
  - 解决:统一了 `run_eval_with_dump.py` 的 dump 格式,所有版本共用同一份验证集与同一套打分逻辑,使得跨版本对比是公平且可追溯的。

### 5.3 对团队协作的客观评价

本组的分工较为清晰:数据组同学负责数据清洗与质量把控,缪梓航同学负责高比例基础题策略的全量 SFT 训练并产出 V3 基模,本人负责微调策略探索、第一阶段全量 SFT 与第三阶段 LoRA 微调。三者形成"数据 → 基模 → 精修"的清晰流水线,各环节产出可独立验证、互不重叠。在两次关键节点(第一次 j1 模型产出、最终 v5 模型产出)上,本人的工作均是端到端独立完成,有完整的脚本、日志与模型产物作为支撑。

### 5.4 收获

本次项目让我对**小参数量模型在限定数据下的微调策略**有了更系统的认识 —— 全量 SFT 适合奠定整体能力基础,而 LoRA 在已有较强基模时是更可控、更不易破坏既有能力的精修工具;数据的"针对性"远比数据的"数量"更重要;每一次评测必须是逐题可追溯的,否则任何"提升"都缺乏说服力。
